# Interview Record: Foundation

Foundation:

- FOUNDATION-001 Brand System
- FOUNDATION-002 Design System

Status: Locked

## Interview Goal

Record completed Foundation decisions so the EmberSoul Labs brand system and design system can guide future specifications without guessing.

## Completed Topics

### Topic 001: Company

User Decisions:

- Company is EmberSoul Labs.

### Topic 002: Tagline

User Decisions:

- Tagline is Ignite the Soul of Every Small Business.

### Topic 003: Brand Positioning

User Decisions:

- Brand positioning is: Building practical AI that helps people live and work more easily.

### Topic 004: Brand Belief

User Decisions:

- Brand belief is: AI should adapt to people, not people to AI.

### Topic 005: Product Philosophy

User Decisions:

- Product philosophy is: Make complex business workflows simple through thoughtful software and AI.

### Topic 006: Product Naming

User Decisions:

- Independent products follow Ember + One English Word.
- Examples include EmberOS, EmberCore, and EmberMind.

### Topic 007: Module Naming

User Decisions:

- Modules are not independent products.
- Examples inside EmberOS include Business Profile, Campaign, Video Studio, CEO Agent, and Marketing Package.

### Topic 008: Target Users

User Decisions:

- Primary target users are Small Business, Retail, Florist, Cafe, Restaurant, Agency, Service Business, Freelancer, and Creator.

### Topic 009: AI Visibility

User Decisions:

- Users know AI is working.
- Users should not need to know provider, model, temperature, or tokens.

### Topic 010: Human Control

User Decisions:

- AI never makes final business decisions.
- Users approve.
- AI executes.

### Topic 011: Automation Principle

User Decisions:

- Users make the decision.
- AI executes the workflow.
- AI follows the latest user-approved instruction until changed.
- AI never assumes changes to user intent.

### Topic 012: Core Design Principles

User Decisions:

- AI should adapt to people, not people to AI.
- Software should deliver value before requiring training.
- Users should learn their business, not the software.
- Good design makes the next step obvious.
- Software should guide users, not test them.
- Keep common tasks simple while keeping advanced capabilities available.
- Users own the intent. AI owns the execution.
- Consistency builds trust.

### Topic 013: Brand Visual Identity

User Decisions:

- Brand Canvas is Dark Professional Canvas.
- Brand Gradient is Blue -> Cyan -> Gold.
- Primary Brand Symbol is Ember Flame.
- Ember Flame is the primary visual identity across all EmberSoul Labs products.

### Topic 014: Color Philosophy

User Decisions:

- Colors communicate meaning.
- Colors must never be used only for decoration.
- Functional colors are Primary, Success, Warning, Error, and Neutral.

### Topic 015: Motion

User Decisions:

- Motion is minimal.
- Animations only support communication.
- Decorative animations should be avoided.

### Topic 016: Icons

User Decisions:

- Icons support text.
- Primary navigation must include text labels.
- Icon-only primary navigation should be avoided.

### Topic 017: One Primary Action Principle

User Decisions:

- Each page should have only one Primary Action.
- Business Profile primary action is Analyze Business.
- Campaign primary action is Generate.
- Video Studio primary action is Regenerate.

### Topic 018: Card First Architecture

User Decisions:

- Business modules should be separated into reusable Cards.

### Topic 019: Empty State Principle

User Decisions:

- Every Empty State should explain why there is no data.
- Every Empty State should explain what the next step is.
- Every Empty State should provide a CTA.

### Topic 020: Screen Orientation Principle

User Decisions:

- Every screen should answer: Where am I?
- Every screen should answer: What's happening?
- Every screen should answer: What's next?

### Topic 021: Zero Confusion Principle

User Decisions:

- Good design makes the next step obvious.
- Avoid making users guess.

### Topic 022: Cross-Platform Principle

User Decisions:

- One Workspace. Any Device.
- Desktop, Tablet and Mobile provide the same product.
- Only presentation adapts.
- Product capability must not be reduced based on device.
- Layout, Navigation, Grid, Drawer, Typography, and Spacing may adapt.
- Business Logic, Workflow, Features, and Permissions must not change by device.

### Topic 023: Device Independence Principle

User Decisions:

- People change devices.
- Their work should not have to.

### Topic 024: Familiarity Over Creativity

User Decisions:

- Prefer interactions users already understand.
- Do not redesign common interaction patterns without strong justification.

### Topic 025: AI Components

User Decisions:

- Use one consistent AI visual language.
- Use Ember Flame as the AI identity.
- Avoid robot mascots.

### Topic 026: AI Assessment Principle

User Decisions:

- AI evaluates quality.
- AI does not guarantee business outcomes.
- Display the required prediction disclaimer whenever AI assessment or scores are shown.

### Topic 027: AI Suggestions

User Decisions:

- AI must explain recommendations before asking users to accept changes.

### Topic 028: Feedback Principles

User Decisions:

- Success feedback explains what completed.
- Success feedback explains the next step.
- Error feedback explains what happened.
- Error feedback explains what users can do.
- Technical errors should not be exposed to normal users.

### Topic 029: Background AI Principle

User Decisions:

- Background AI jobs must display progress.
- Background AI jobs must display estimated time.
- Background AI jobs must display notifications.
- Users may continue working.

### Topic 030: Never Leave Users Wondering

User Decisions:

- Users should always know whether the request was received.
- Users should always know what the system is doing.
- Users should always know whether they need to wait.
- Users should always know what happens next.

### Topic 031: Never Lose User Work

User Decisions:

- Once users submit a task, the system should protect it.
- If AI services are temporarily unavailable, queue the request.
- Resume automatically once services recover.
- The recommended Request Saved UX copy is accepted as a Product Decision.

## Architect Recommendations

Architect Recommendations are not attributed to the user unless explicitly accepted.

The Never Lose User Work UX copy originated as an Architect Recommendation and was later accepted as a Product Decision.
